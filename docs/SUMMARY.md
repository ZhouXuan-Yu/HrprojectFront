## 一、最终结论（下次对话先看这个）

### 1.1 项目定位
基于 Python Flask 的智能招聘系统，核心链路：**邮箱收简历 → AI解析画像 → 人岗匹配 → 面试(飞书) → Offer → 入职同步**。

### 1.2 最终确认的架构（4 模块，不是 9 模块）

```
① 招聘需求管理    需求单→审批→生成岗位→内外部并行搜索
② 人才库         外部人才库(简历/候选人) + 内部人才库(在职员工招聘维度数据)
③ 招聘流程       匹配打分→人工复核→面试(飞书)→Offer→hire_event→入职同步
④ 招聘数据报表   全链路数据汇总（不建独立表，聚合前三块的消费层）
```

### 1.3 关键设计决策

| 决策 | 结论 |
|------|------|
| 编制校验 | v0.1 先跳过，HR 手动判断 |
| 面试轮次 | 表结构预留 `round` 字段，v0.1 做单轮 |
| 飞书集成 | v0.1 只做建会议+发通知，日历同步放后面 |
| 内部人才库 | v0.1 保留表和流程，但不做先内后外的串行——内外并行 |
| IAM 底座表 | 不删除，不重建——用 hr-master IAM 的 `t_core_org`/`t_core_position`/`t_core_user` |
| 内部搜索 | 系统自动匹配合格线，面试官看结果；但内外并行，永远发布外部渠道 |
| 表命名 | `t_core_*` 跟 IAM，`t_hr_*` 招聘自建 |
| 逻辑外键 | 不建物理 FK，值存 IAM 的 id |

### Day 9（7月15日）：技术细节确认 & 前后端方案定稿

**我通读了 SUMMARY.md + PRD_FINAL.md + 7 个原型页面 + app.js + style.css**，提了 11 个问题逐条确认。

#### 一、Dify AI 对接细节（Q4）

| 工作流 | 触发时机 | 说明 |
|--------|---------|------|
| AI 简历画像解析 | 简历上传 / 邮箱同步 | 读取简历文件 → 抽取基础信息 → 生成技能标签 → 输出结构化候选人数据 |
| 人岗匹配打分 | 需求审批通过自动触发 / 人才库手动匹配 | 传入岗位 JD + 候选人画像 → 多维度计算匹配分 + 生成匹配说明 |
| 面试问题生成 | 面试页 AI 话术按钮 / 飞书智能对话 | 传入 JD + 简历 + 面试轮次 → 分阶段输出面试题库 |

**API 调用规则**：统一后端代理转发，禁止前端直连 Dify。
链路：`前端操作 → 本系统后端接口 → 后端携带密钥调用 Dify → 后端清洗数据返回前端`。
好处：保护密钥、校验数据权限、限流缓存、统一捕获接口异常。

#### 二、多租户 tenant_id（Q5）

- **作用**：SaaS 标准数据隔离字段，业务表全部预埋，租户间完全隔离
- **v0.1**：单租户起步，全表预留 `tenant_id` 字段统一默认值，底层查询逻辑兼容租户过滤，不做租户切换/管理后台
- **后续**：仅新增租户管理模块，无需重构表、迁移数据

#### 三、核心状态机（Q6）

`t_hr_recruit_process` 状态流转：

```
候选人入库 → 待筛选 → 待安排 → 已安排 → 进行中 → 待评价 → 已评价
                                                      ├─ 通过 → 发Offer → 已录用
                                                      └─ 不通过 → 回流人才库
```

关键澄清：**「已锁定」不是 process 表状态，而是 `t_hr_candidate.status`**。候选人进入 process（待安排~已评价期间）→ 候选人状态 = 进行中(锁定)，防止被其他需求重复发起。两张表联动变更，process 完结后自动释放锁定。

#### 三、匹配打分设计

**打分逻辑**：
```
最终得分 = 时效分 × 30% + 匹配分 × 70%（权重可配）

时效分：简历入库时间 → 衰减曲线（今天100→30天70→90天30）
匹配分：JD技能标签 × 候选人标签 → 命中加分/缺失扣分
```

**两种触发模式**：
- 自动：需求审批通过 → Celery 异步 → 批量匹配全部候选人 + 内部员工 → 写入匹配表
- 手动：HR 在 AI中心/需求管理点击匹配 → 实时调用 Dify 工作流② → 返回排序结果

**按岗位展示打分（不按候选人汇总）**：
```
需求管理
├── 岗位：高级Java工程师
│   ├── 张三 85分 🟢  → 约面
│   ├── 孙九 68分 🟠  → 待定
│   └── 李四 55分 ⬜  → 不合适
├── 岗位：前端工程师
│   ├── 郑一 88分 🟢
│   └── 张三 55分 ⬜  ← 同人不同岗不同分
```

**展示分层**：
- 需求管理列表列 → 只显示匹配人数（如 `内部匹配 2人（王工92·赵工42）`）
- 需求详情 → 各维度明细 + AI 打分理由
- AI 中心 → 雷达图 + 候选人对比 + 调分锁定
- 候选人抽屉 → 画像评级 + 匹配标签（命中/缺失）

**分数线**：
- `pool_min_score = 60` → 简历库准入线
- `auto_invite_min = 80` → 自动邀约线
- 颜色：≥80 绿 / 60-79 橙 / <60 灰

### Day 10 补充（7月15日）：模块边界最终确认 + 画像分 vs 匹配分 + 推荐流水线

#### 一、核心澄清：两个分，不一样

| | 人才画像分 | 岗位匹配分 |
|------|-----------|-----------|
| 算什么 | 这个人本身能力怎么样 | 这个人适合这个岗位吗 |
| 依据 | 学历、学校、年限、公司背景、证书 | JD 技能标签 × 候选人技能标签 |
| 举例 | 研究生+2分、985+1分、大厂+1分 | Java权重5×命中=+5、K8s权重3×缺失=-3 |
| 存哪里 | `t_hr_resume.base_score` | `t_hr_resume_match.match_score` |
| 哪里展示 | **人才库**列表 + 候选人抽屉 | **需求管理**详情页（按岗位分组） |
| 谁算 | Dify WF① 简历解析时一次性算 | Dify WF② 每次匹配时算 |
| 谁看 | HR 扫简历时参考 | HR 做岗位筛选决策时参考 |

**画像分跟着人走，匹配分跟着岗位走。两者不混。**

#### 二、人才库最终定位

人才库 = 候选人的档案室。四件事：存、看、管、提供数据。

**列表展示**：编号、姓名、画像（A/B+/C+ + 画像分）、学历、年限、核心技能（前3标签）、最近公司、来源、状态、操作。

**列表不展示**：岗位匹配分（没有岗位上下文）、约面按钮（不是人才库的职责）。

**操作按钮**：匹配岗位（选岗→跳需求详情）、查看（抽屉）、编辑标签。约面从需求详情发起。

**候选人抽屉**：基本信息 → AI画像 → 结构化简历 → 技能标签云 → **匹配记录按岗位展示** → 面试记录（只读） → 状态时间线 → 原始简历。

#### 三、岗位发布后的完整推荐流水线

```
岗位审批通过
  │
  ├── ① 系统自动扫简历库
  │     └── Dify WF② 逐个打分
  │         └── 按匹配分排序
  │             ├── Top N 进「系统推荐」
  │             └── 其余进「全部匹配结果」（可查，不放推荐）
  │
  ├── ② 外部直接投递
  │     └── 绑定 demand_id
  │         └── 系统后置算一个匹配分
  │             └── 同样筛：高分进推荐，低分可查
  │
  └── ③ 没进推荐的
        └── 回流简历库
            └── HR 可手动翻、手动搜
            └── 不放推荐列表
```

### Day 11（7月16日）：评分体系定稿 + 前端全面整改 + 面试权限设计 + 录制存储方案

#### 一、评分体系最终定稿

**三个分，各司其职：**

| 分 | 算什么 | 算法 | 存哪 | 展示在哪 |
|----|--------|------|------|---------|
| 画像分（外部） | 学历+院校+工龄+大厂+证书 | 硬规则固定加分 | `t_hr_candidate` | 人才库列表 |
| 综合评估（内部） | 绩效+工龄+晋升+培训 | 硬规则，跟外部算法不同 | `t_hr_employee` | 内部员工库 |
| 岗位匹配分 | JD标签×候选人标签命中率 | Dify AI 每次匹配实时算 | `t_hr_resume_match` | 需求详情 |

**综合推荐公式：**
- 直接投递：`画像分 × 10% + 原始匹配分 × 90%`（无衰减）
- 存量检索：`画像分 × 10% + (原始匹配分 × 衰减系数) × 90%`
- 衰减系数：≤30天=1.0，30-90天=0.85，>90天=0.70

衰减系数只影响推荐排序，不改数据库原始匹配分。超30天黄色提醒「⚡匹配分已超期，建议重新匹配」。
