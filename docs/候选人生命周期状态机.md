# 候选人生命周期状态机

> 一页纸说明人才库、需求管理、面试计划、录用入职四个模块如何联动。
> 核心原则：**一个候选人在同一需求下只有一条"进行中"的招聘流程（RecruitProcess），各模块读写同一条流程状态，人才库候选人状态随流程自动变化。**

## 1. 全景图

```
人才库                需求管理                  面试计划              录用/入职
─────────────       ──────────────────       ──────────────       ──────────────
可联系 ──加入需求──▶ 流程0·待筛选
                     (+ JD匹配分写入 t_hr_resume_match)
面试中(锁定) ◀────── 流程2一面/3二面 ◀────── 发起面试自动建/挂流程
可联系(解锁) ◀────── 流程4·淘汰 ◀────────── 评价淘汰 / 取消面试
                     流程0·待筛选 ◀──────── 淘汰/放弃后允许重新加入
面试中(锁定)          流程5·待Offer ◀─────── 评价通过
                      流程6·已接受 ◀─────── 候选人点确认链接接受 Offer
已入职 ◀──────────── 流程8·入职 ◀────────── 确认入职（自动建入职单+7项清单）
可联系(解锁) ◀────── 流程7·放弃 ◀────────── Offer 被拒/撤回/候选人拒绝
```

## 2. 状态字典

### 流程状态 `t_hr_recruit_process.process_status`

| 值 | 含义 | 触发动作 |
|---|---|---|
| 0 | 待筛选 | 加入需求 / 淘汰后重新加入 |
| 1 | 已邀约 | （预留）邀约候选人 |
| 2 | 一面中 | 发起初试面试 |
| 3 | 二面中 | 发起复试/终面 |
| 4 | 淘汰 | 评价 fail / 取消面试 |
| 5 | 待Offer | 评价 pass |
| 6 | 已接受 | 候选人接受 Offer（确认链接或 HR 代操作） |
| 7 | 放弃 | Offer 被拒 / 撤回 / 候选人拒绝 |
| 8 | 入职 | 确认入职 |

### 候选人状态 `t_hr_candidate.status`

| 值 | 含义 | 何时设置 |
|---|---|---|
| `available` | 可联系 | 入库；淘汰/取消/Offer 终止后自动解锁回来 |
| `locked` | 面试中(锁定) | 发起面试时自动锁定；锁定期不可重复加入需求 |
| `reserve` | 储备 | HR 手动标记 |
| `archived` | 已封存 | HR 手动归档 |
| `hired` | 已入职 | 确认入职时自动设置 |

## 3. 联动规则（按动作）

### 3.1 加入需求 `link_candidate_to_demand`
- 写入**真实 resume_id**（候选人最新一份简历），不留 0
- 黑名单（`black_flag=1`）、`locked` 状态候选人拒绝加入
- 去重：同候选人 + 同需求存在进行中流程（状态 ∉ {4,7}）时不重复创建，返回 `already`
- 淘汰（4）/放弃（7）后允许重新加入，创建新流程
- 加入时用 JD 对简历解析结果做 AI 匹配打分，写入 `t_hr_resume_match`

### 3.2 发起面试 `create_interview`
- 按简历定位候选人，按 (candidate_id, demand_id) 查进行中流程；没有则创建（状态 0）
- 流程状态推进：初试 → 2，复试/终面 → 3
- 候选人 `status='locked'`，人才库同步显示"面试中(锁定)"
- `InterviewBook.process_id` 写真实值，评价/取消时沿此更新流程

### 3.3 面试评价 `evaluate_interview`
- 评价理由必填（≥5 字），不可仅点通过/拒绝
- `pass` → 流程 5（待Offer）；`fail` → 流程 4（淘汰）+ **解锁候选人**回 available
- `hold` → 保持面试中，不改流程
- 已评价的面试不可重复评价

### 3.4 取消面试 `cancel_interview`
- 已通过的面试不可取消
- 流程 → 4，候选人解锁回 available

### 3.5 Offer 生命周期 `hire_service`
- 发 Offer → 流程 5；候选人接受 → 流程 6 + **自动建入职单（Entry + 7 项清单）** + 发入职包邮件
- Offer 被拒 / 撤回 / 候选人通过确认链接拒绝 → 流程 7 + 需求编制 `filled_count` 回退 + **解锁候选人**回 available

### 3.6 确认入职 `confirm_onboard`
- 流程 → 8，候选人 `status='hired'`，人才库显示"已入职"

## 4. 关键实现位置

| 功能 | 文件 | 函数 |
|---|---|---|
| 加入需求 + 匹配打分 | `backend/app/services/demand_service.py` | `link_candidate_to_demand` |
| 需求详情候选人视图 | 同上 | `list_demand_candidates`（优先读真实匹配分） |
| 面试创建联动 | `backend/app/services/interview_service.py` | `create_interview` / `_ensure_process_for_interview` |
| 解锁辅助 | 同上 | `_release_lock_for_resume`（resume_id → candidate_id） |
| 评价/取消/入职 | 同上 | `evaluate_interview` / `cancel_interview` / `confirm_onboard` |
| Offer 状态机 | `backend/app/services/hire_service.py` | `create_offer` / `accept_offer` / `reject_offer` / `withdraw_offer` |
| 候选人确认链接 | `backend/app/services/confirm_service.py` | accept/reject 回流走同一套 hire_service |
| 人才库状态标签 | `backend/app/services/talent_service.py` | `STATUS_LABELS` |

## 5. 注意事项

- **匹配分只是推荐度建议，绝不阻断动作**：不存在"匹配分不足"状态。低分（<60）或有 `notRecReason` 的候选人，UI 仅显示"建议不推荐"提示（悬停见原因），联系和约面按钮始终可用，HR 可自主决定是否推进
- **解锁必须走 resume_id → candidate_id 转换**：面试/Offer 表存的是 resume_id，直接拿它当候选人 ID 会解错对象（历史 bug，已修，勿回退）
- **业务编号唯一性**：RP/OF/HE/EN 编号 = 前缀 + 秒级时间戳 + 3 位随机数，同秒连续创建不撞键
- **匹配分展示**：`list_demand_candidates` 优先取 `t_hr_resume_match` 最新真实打分，无记录时才退回确定性估算（`match_service.calc_match_score`）
- **验证基线**：全链路 7 场景 28 项断言曾端到端跑通（加入→面试→淘汰→重加入→通过→Offer→入职→拒 Offer→取消），改动联动逻辑后应按同路径回归
