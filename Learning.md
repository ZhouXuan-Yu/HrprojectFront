# Learning.md - 复盘记录

> 记录这次哪里做错了、为什么错、下次如何改。这里不写流水账，只写可复用的教训。

## 复盘 #1 - 未经确认删除用户文件

### 问题

早期整理文件夹时，曾把根目录的资料和目录删除，以为“已经复制到新目录就可以清理原件”。

### 为什么错

- 用户要求新建或整理，不等于授权删除。
- 删除、移动、覆盖是高风险操作。
- Windows 下文件锁可能导致删除执行一半，恢复成本高。

### 下次如何改

- 默认只新增或精确修改。
- 删除、移动、清空目录前必须明确询问。
- 不使用破坏性命令处理用户资料。

## 复盘 #2 - 跨目录复制后忘记路径差异

### 问题

把 HTML 从一个目录复制到另一个目录后，曾忘记 CSS/JS 引用路径差异，导致页面资源失效。

### 为什么错

- `prototype/` 和 `hr-web/frontend/` 的目录结构不同。
- 同名 HTML 在不同目录下依赖路径不同。

### 下次如何改

- 只要移动 HTML，就检查 `<link>`、`<script>`、图片和跳转路径。
- 复制后必须打开目标页面或跑测试。

## 复盘 #3 - UI 方案没先收敛，导致反复返工

### 问题

早期在漏斗、预警、备注等 UI 上，直接写代码后再等用户纠正，导致同一功能反复修改。

### 为什么错

- UI 是判断密集型任务，用户对视觉有明确偏好。
- 未先说明设计方向，容易做成“能运行但不喜欢”。

### 下次如何改

- 视觉改造先用 2-3 句话说明方向。
- 用户已经表达“不喜欢”时，先归纳不喜欢的点，再改代码。
- 同一方向连续返工两次后，要停下来重新确认方案。

## 复盘 #4 - 登录页一度太像 AI 动效 demo

### 问题

登录页最初加入猫头鹰/角色式眼睛跟随动效，虽然有趣，但用户反馈“AI 味道太重”，不符合专业企业级系统。

### 为什么错

- 把“动态有趣”放在了“企业可信”前面。
- HR 系统登录页需要稳定、清晰、专业，动效只能做品牌记忆点。

### 下次如何改

- 企业系统优先使用克制、信息化、流程化视觉。
- 动效要解释业务语境，例如人才雷达、流程节点、状态反馈。
- 避免动物、拟人、过度可爱、过度科技口号。

## 复盘 #5 - 参考 GitHub 设计时需要版权隔离

### 问题

用户指出参考仓库设计不错，但担心版权问题，要求做专属于本项目的设计。

### 为什么错

- “参考”不能变成“复刻”。
- 即使没有复制文件，只要角色形态、布局、配色和交互组合过于接近，也会有争议。

### 下次如何改

- 只抽象通用交互原理，不复制视觉表达。
- 明确写出原创化边界：不复制源码、素材、文案、构图、角色比例。
- 用项目业务语言重构视觉：本项目采用“人才雷达中控台”。

## 复盘 #6 - 长期文档没有及时纳入追踪

### 问题

`Claude.md`、`Memory.md`、`Learning.md`、`Wiki.md` 曾出现在未跟踪状态，说明项目记忆没有真正进入版本控制。

### 为什么错

- 用户需要长期追踪，未纳入 Git 就无法稳定复用。
- 后续 agent 可能看不到最新约定。

### 下次如何改

- 用户确认四文档制度后，四个文档必须纳入 Git。
- 每三轮实际修改更新一次 `Memory.md`。
- 出现纠正、返工、测试失败时，立即更新 `Learning.md`。

## 复盘 #7 - AI 外呼口径不符合实际业务

### 问题

多个页面曾出现“AI 外呼”“自动拨打电话”等表达。用户明确指出这不符合实际，应该去掉或换成其他方式。

### 为什么错

- 当前系统是前端原型，没有真实电话能力。
- 招聘联系候选人属于敏感沟通，不能暗示系统会自动打电话或替 HR 完成实时沟通。
- “AI 外呼”会让企业级产品显得不可信，也容易误导后续研发范围。

### 下次如何改

- AI 只能表述为辅助：生成联系话术、整理沟通摘要、提示缺失信息、生成下一步建议。
- 实际联系方式写成电话、邮件、飞书、联系直属上级。
- 约面前的联系确认必须是人工确认结果，不写“已 AI 外呼”。
- E2E 中保留“页面不得出现外呼/自动拨打”的断言。

## 规则沉淀

| 规则 | 说明 |
|---|---|
| 不删用户文件 | 删除、移动、覆盖必须先确认 |
| 先确认视觉方向 | UI 不是只看能不能跑 |
| 参考不复刻 | GitHub 设计只能抽象交互原理 |
| 企业级优先 | 少卡通、少口号、少 AI 味 |
| AI 不代替电话 | 候选人联系只能由 HR 通过真实渠道确认 |
| 测试后交付 | build、E2E、截图至少覆盖对应风险 |
| 三轮一更新 | 每三轮实际修改同步长期文档 |

## 复盘 #8 - 响应式不能只靠桌面 E2E

### 问题

UI v6 改造后，桌面和功能测试均通过，但移动端视觉巡检发现看板页存在横向溢出：未包裹在 `.table-wrap` 中的普通表格继承了 960px 最小宽度，撑开了移动页面。

### 为什么错

- E2E 功能测试主要验证可见性和交互，没有检查移动端 `scrollWidth`。
- 全局表格 token 对标准 `.table-wrap` 有效，但遗留页面中仍存在直接放在 `.card` 内的表格。
- 响应式修复第一次复查时使用了旧的 `dist` 预览包，未先重新 build，导致误判。

### 下次如何改

- UI 改造后必须跑桌面 + 移动截图巡检，并检查 `document.body.scrollWidth <= innerWidth`。
- 修改 CSS 后若使用 `npm run preview`，必须先 `npm run build`。
- 遗留页面里凡是直接表格都要么补 `.table-wrap`，要么在移动端让父容器 `overflow-x:auto`。

## 复盘 #9 - Legacy 外壳增强必须等待页面脚本注入完成

### 问题

Phase 2 新增工作台外壳增强时，第一次只在全局脚本初始化阶段执行，Playwright 发现 `.content` 没有补上 `workbench-ready`。修正后又发现命令按钮没有随 legacy 页面注入完成一起安装。

### 为什么错

- 当前项目是 Vue shell 承载 legacy HTML，真实侧栏和顶栏由每个 legacy 页面内联脚本调用 `renderSidebar()` 后才出现。
- 全局脚本初始执行时，DOM 外壳可能还不完整。
- 只暴露“增强结构”函数不够，Vue 路由切换后需要重新执行完整外壳安装。

### 下次如何改

- 凡是依赖 legacy 页面 DOM 的增强，都必须在 `LegacyPage.vue` 完成 HTML 注入、内联脚本执行、链接归一化之后再执行。
- 暴露给 Vue shell 的函数必须是完整安装入口，而不是半成品增强函数。
- E2E 要覆盖结构状态和真实点击路径，不能只覆盖键盘快捷键。

## 复盘 #10 - 数据可视化后台不能只做 token 化

### 问题

Phase 3 初始思路偏向“全局增强补丁”，但用户指出几个页面 UI 仍然太 low，不符合千亿级项目的数据可视化页面标准，并提供了 `D:\WorkProject\HeroUIPro\herouipro-v3` 作为组件参考源。

### 为什么错

- token、边框、圆角只能解决一致性，不能自然形成专业数据产品的组件语言。
- 数据页需要成熟结构：KPI 的 title/value/trend/context，表格的 Filter/Sort/Columns/Search，图表卡的标题、主数、趋势、legend、口径。
- 当前项目是 legacy HTML，不代表只能停留在低质表格和普通卡片；可以通过渐进增强吸收 HeroUIPro 的组件范式。

### 下次如何改

- 做后台组件前先检查 `D:\WorkProject\HeroUIPro\herouipro-v3`。
- 能复制的复制，不能直接复制的抽取结构、状态和 CSS 语法。
- UI 验收不仅看是否统一，还要看是否像真正的数据工作台，尤其是表格、KPI、图表和筛选的专业表达。

## 复盘 #11 - 参考图要转译成组件系统，而不是停留在灵感描述

### 问题

用户继续反馈“当前网页设计太低级”，并补充多张参考图。之前虽然已经查看 `D:\WorkProject\HeroUIPro\herouipro-v3`，但第一轮落地更多是增强已有组件，没有足够把参考图中的页面层结构转成可复用模式。

### 为什么错

- 高级后台的质感来自页面结构：模块导航、KPI 摘要、图表卡、表格密度、状态层级一起工作。
- 只修单个表格或 token，会让页面仍像普通 CRUD 原型。
- 用户说“可以直接复用组件代码”时，意味着要优先找成熟组件模式并落地到项目，而不是重新解释设计理念。

### 下次如何改

- 遇到后台 UI 提升任务，先从 HeroUIPro 中找相近组件，再判断能否复制、改写或抽象成当前项目增强器。
- 页面级改造必须至少包含：导航层、指标层、主数据层、辅助分析层和移动端策略。
- 每轮 UI 改造必须用截图肉眼复审，不能只看自动化通过。

## 复盘 #12 - Legacy 页面 head 样式不会自动进入 Vue shell

### 问题

招聘看板的折叠栏看起来有 `.collapse-body{display:none}`，但在 Vue shell 下内容区仍然可见，导致折叠栏不能正常折叠。

### 为什么错

- 当前 `LegacyPage.vue` 注入的是 legacy 页面 body 内容，页面 `<head>` 内的局部 `<style>` 不会作为运行样式进入应用。
- 折叠交互依赖内联 `onclick` 和页面 head 样式，结构脆弱；一旦进入 Vue shell，就出现状态和可见性不一致。
- 之前测试只覆盖了 tabs、弹窗、提醒，没有直接断言招聘看板折叠栏。

### 下次如何改

- legacy 页面中会影响交互可见性的样式必须进入全局 CSS 或组件 CSS，不能只放在 HTML head。
- 所有折叠/手风琴交互必须有 `aria-expanded`、键盘操作和显隐断言。
- UI 规则如“禁止渐变色”要用全项目搜索验证，不只看当前修改区域。
